## 2.2 New tables

### A) `change_request_approvals`
- Purpose: keep each approval stage decision as immutable records.

| column | type | note |
|---|---|---|
| id | bigserial pk | |
| change_request_id | bigint fk -> change_requests(id) ON DELETE CASCADE | |
| stage | approval_stage | TEAM_LEAD / ADMIN |
| step_order | integer | 1, 2... |
| approver_user_id | bigint fk -> users(id) ON DELETE SET NULL | nullable until acted |
| status | approval_status | default PENDING |
| comment | text | rejection/approval reason |
| acted_at | timestamptz | when approved/rejected |
| created_at | timestamptz default now() | |
| updated_at | timestamptz default now() | |

Recommended constraints/indexes:
- `UNIQUE(change_request_id, stage)`
- `INDEX(change_request_id, step_order)`
- `INDEX(status, stage)`

### B) `evaluation_jobs`
- Purpose: async worker queue in DB for submission evaluation.

| column | type | note |
|---|---|---|

| status | evaluation_job_status | default QUEUED |

| max_attempts | integer default 3 | |
| queued_at | timestamptz default now() | |
| started_at | timestamptz | |

Recommended constraints/indexes:

## 2.3 Existing table changes

### `change_requests`
- Add `current_stage approval_stage` (default `TEAM_LEAD`)
- Add `final_decision_at timestamptz` (nullable)

### `submissions`
- Add `change_request_version_snapshot integer` (nullable)
  - On create with `change_request_id`, copy current `change_requests.version`.
  - Keep this snapshot immutable for audit safety.
- Add `evaluation_status evaluation_job_status` (nullable)
  - Derived/cached status for fast UI list rendering.

Notes:
- Keep existing `change_request_id` to preserve traceability.
- Prefer `ON DELETE RESTRICT` for `change_request_id` in future migration if data retention policy requires immutable history.

## 3) API Delta (Draft)

All endpoints below are additive; existing MVP endpoints remain valid.

## 3.1 Change request approval chain

### `GET /api/change-requests/:changeRequestId/approvals`
- Auth: employee(owner) or admin
- Response:
```json
{
  "changeRequestId": 101,
  "currentStage": "TEAM_LEAD",
  "approvals": [
    { "stage": "TEAM_LEAD", "status": "PENDING", "approverUserId": null, "comment": null, "actedAt": null },
    { "stage": "ADMIN", "status": "PENDING", "approverUserId": null, "comment": null, "actedAt": null }
  ]
}
```

### `POST /api/change-requests/:changeRequestId/approve`
- Auth: approver role by stage
- Body:
```json
{ "comment": "Looks good" }
```
- Behavior:
  - Approve current stage
  - Advance `current_stage` or finalize as `APPROVED`

### `POST /api/change-requests/:changeRequestId/reject`
- Auth: approver role by stage
- Body:
```json
{ "comment": "Need scope clarification" }
```
- Behavior:
  - Reject current stage
  - Finalize as `REJECTED`
  - `comment` required

### `GET /api/admin/change-requests/pending`
- Query: `stage=TEAM_LEAD|ADMIN`, `ownerEmployeeId`, `department`, `q`
- Purpose: inbox view by approval stage

## 3.2 Evaluation queue / result

### `POST /api/submissions/:submissionId/finalize` (existing extension)
- Existing finalize remains.
- New behavior:
  - Create/update `evaluation_jobs` as `QUEUED`
  - Set submission/work item status to `EVALUATING` (optional policy switch)

### `GET /api/submissions/:submissionId/evaluation`
- Auth: owner or admin
- Response:
```json
{
  "job": {
    "status": "RUNNING",
    "queuedAt": "2026-03-02T00:00:00.000Z",
    "startedAt": "2026-03-02T00:00:05.000Z",
    "finishedAt": null,
    "attemptCount": 1
  },
  "latest": null
}
```

When done:
```json
{
  "job": { "status": "SUCCEEDED", "queuedAt": "...", "startedAt": "...", "finishedAt": "..." },
  "latest": {
    "evalVersion": 1,
    "totalScore": 86.5,
    "grade": "A",
    "summaryText": "Strong execution and evidence quality.",
    "checklist": [],
    "evidence": []
  }
}
```

### `POST /api/admin/submissions/:submissionId/evaluation/retry`
- Auth: admin
- Body:
```json
{ "reason": "Model timeout on prior run" }
```
- Behavior:
  - Queue new evaluation attempt (increment version on success)

## 4) Worker Contract (Draft)

Worker loop (DB polling MVP):
1. `SELECT ... FOR UPDATE SKIP LOCKED` one `evaluation_jobs` row with `QUEUED`
2. Mark as `RUNNING`, set `started_at`, `worker_id`
3. Load submission file metadata + context
4. Run evaluator (LLM or rules)
5. Insert `evaluations` row (`eval_version = previous + 1`)
6. Mark job `SUCCEEDED`, set `finished_at`
7. Update submission/work item status to `DONE` (policy dependent)

Failure path:
- Increment `attempt_count`
- If attempts left: back to `QUEUED`
- Else: `FAILED` and keep `error_message`

## 5) Migration Sequence (Draft)

1. Add new enums/tables (`change_request_approvals`, `evaluation_jobs`, `evaluations`)
